# 3エージェント開発ハーネス 手順書

ストップウォッチアプリ開発で実践した「Planner → Generator → 人間レビュー」方式の進め方をまとめたもの。アプリ・ツール開発全般で再利用する。

実際の進行は `/sprint` コマンド（`~/.claude/commands/sprint.md`）に定義されており、そちらが正となる。このドキュメントは考え方と背景の解説。

## 全体の流れ

```
フェーズ0: 仕様書づくり・スプリント分割（人間と一緒に固める。コードは書かない）
フェーズ1: Planner が詳細仕様を作る
フェーズ2: Generator が worktree 隔離で実装する
フェーズ3: Claude本体がレビュー → main に反映 → push
フェーズ4: 実機/ブラウザで動作確認 → フィードバック → 次のスプリントへ
```

各フェーズには完了条件があり、満たすまで次に進まない。

## 各エージェントの役割

### ヒアリング・スプリント分割（人間 ⇄ Claude本体）

- 最初に1〜4行の要望を聞き、曖昧な点を質問で潰す（実装方法ではなく「何を作るか」のレベル）
- 仕様書（docs/xxx-spec.md）としてまとめる。仕様が固まるまでコードは書かない
- 仕様書を作業単位のスプリントに分割し、進捗表を docs/ に保存する

### Planner（Agent: subagent_type="Plan"）

- 役割: スプリントの詳細仕様を「何が起きるか」レベルで展開する
- **技術的な実装詳細（データ設計、関数名など）には踏み込ませない**
  - 理由: Plannerが実装方法を間違えると下流（Generator）に伝播するため
- 入力: スプリントのスコープ、現在のコードの状態
- 出力: 詳細な製品仕様書（状態遷移、UI、データの振る舞いなど）
- 仕様に実装詳細が混ざっていたら、Claude本体が削ってから使う（再依頼は不要）

### Generator（Agent: isolation="worktree"）

- 役割: Plannerの仕様を実際のコードに実装する
- 必ず `isolation: "worktree"` で隔離実行する（本体に影響を与えない）
- Generatorは過去の会話を覚えていないため、プロンプトは自己完結させる。既存コードの構造（変数名・関数名の実例を5〜10個）を明記し、一貫性を保たせる
- 完了後、Claude本体がworktreeの差分を読んでレビューする

### 人間によるレビュー・動作確認

- Claude本体がworktreeのコードをReadで確認し、問題の程度で対応を分ける
  - 軽微 → 本体が直接修正して取り込む
  - 仕様とのズレ → Generatorに具体的に指摘して再依頼
  - 差分が巨大で読み切れない → 取り込まず、スプリントを分割してやり直す
- 問題なければmainにコピー → コミット → push（確認不要。実機確認のため常に最新を保つ）
- 実機（iPhone等）やブラウザで動作確認してもらい、フィードバックを反映する

## スプリント分割の基準

**1スプリント = 1作業。** 「1作業」と言えるかは以下で判定する。

- 変更が1つの関心事に閉じている（「タブバーの見た目」「開始ボタンの挙動」「設定値の保存」はそれぞれ別作業）
- 人間が1〜2分で動作確認できる
- レビューする差分が一度に読み切れるサイズ（目安: 200行以内）
- スコープの説明に「AとB」のように「と」が入ったら分割を検討する

作業単位まで細かくするとスプリント数が増えるため、「グループ（機能）→ スプリント（作業）」の2階層で管理する。

| グループ | # | 作業内容 | 状態 |
|---------|---|---------|------|
| 基本UI | 1-1 | タブバーの見た目 | 完了 |
| 基本UI | 1-2 | メイン画面レイアウト | 作業中 |
| 計測 | 2-1 | 開始・停止の動作 | 未着手 |

迷ったら小さく割る。細かすぎる失敗は取り返しがつくが、大きすぎる失敗はレビュー不能な差分を生む。

## 失敗パターンと対処

| 症状 | 対処 |
|------|------|
| Plannerが実装詳細を書いてくる | 本体が仕様からその部分を削除して使う |
| Generatorが命名規則を無視する | プロンプトの命名実例が少なかったサイン。実例を増やして再依頼 |
| Generatorの差分が巨大 | スプリントが大きすぎる。取り込まず分割し直す |
| worktreeの変更が本体と競合する | 本体側の未コミット変更を先にコミットしてから再実行 |
| 実機でだけ動かない | 原因が判明したら、注意点をプロジェクトのCLAUDE.mdに追記して次回に活かす |

## 実践のコツ

- **Plannerに実装詳細を書かせない**: 仕様が技術選定を縛ると、Generatorの自由度が下がり、下流で帳尻合わせが必要になる
- **Generatorへの指示は具体的に**: 既存の変数名・関数名・コード構造を明記する。そうしないとGeneratorが命名規則を変えてしまい、スプリント間で一貫性が崩れる
- **worktree隔離は必須**: Generatorの実装ミスが本体のコードに直接影響しないようにする
- **実機確認は早めに**: ブラウザのプレビューツールが使えない環境（サーバー起動不可など）もあるため、GitHub Pages等で実機から直接確認できる経路を早めに作る
- **アプリ固有の規約はCLAUDE.mdに置く**: 難読化ルールやプラットフォーム固有の注意点はプロジェクトのCLAUDE.mdに集約し、ハーネス自体は汎用に保つ

## このプロジェクトでの実例

ストップウォッチアプリ（`stopwatch.html`）を以下のスプリントで開発した（当時は1スプリント=1機能の粒度。現在の基準ではさらに作業単位に分割する）。

| Sprint | 内容 |
|--------|------|
| 1 | 基本UI（タブバー＋画面の見た目） |
| 2 | 開始・停止・リセットの動作 |
| 3 | ラップ機能 |
| 4 | コア機能（コア計算ロジック） |
| 5 | 隠し設定画面 |
| 6 | PWA対応（アイコン・manifest） |
| 7 | UI仕上げ |
| 追加 | 開始トリガー設定の追加機能 |

電卓フォースの開発でも同じハーネスを使用した。
