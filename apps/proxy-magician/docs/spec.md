# 代打マジシャン（毒舌AIマジシャン）仕様書（ドラフト v0.2）

> マジックバーで接客中、マジシャンが数分離席する「つなぎの3分間」を成立させるためのスマホマジックアプリ。
> スマホの毒舌AIマジシャンがお客様の言ったカードを当て、最後に物理のトランプケースからも同じカードが出る。

> **v0.2 変更点**：物理リビールの基準を「AIが言ったカード」→「お客様が実際に言ったカード」に変更（タネの根幹）。
> これにより音声認識はトリックの必須要素ではなくエンタメ演出扱いになり、技術リスクが大幅に低下。

---

## 1. 目的・利用シーン

- マジシャンがトイレ等で **3分だけ離席**する間、卓を持たせる（お客様を飽きさせない・接客の空白を埋める）
- スタート後は **ハンズフリーで自動進行**する（マジシャンはその場にいない前提）
- 一発ネタとして完結し、マジシャンが戻ってきて物理リビールでオチをつける

## 2. タネの構造（最重要・機密）

お客様に見える見せ場は2段階だが、マジック的な重みが全く違う。

| ステップ | 見た目 | 実態 |
|---|---|---|
| ④ 画面内でカードが当たる | すごそう | **タネなし。** スマホなら当然できる（お客様も「スマホやから」と思う） |
| ⑥ 物理ケースから本物が出る | 「スマホやろ」と油断した所に一撃 | **これが本当のマジック** |

### 核心：物理リビールは「お客様が実際に言ったカード」に合わせる

- 物理の当ては **インビジブルデック**で行う（自由に言われた1枚を出せる）
- マジシャンは戻ってきたら、**お客様に本当のカードを確認して**、そのカードをインビジブルデックで出す
- **AIが画面に出したカードには合わせない。**
  - 理由：AIが言ったカードを物理で出すと「最初からAIがそのカードを言うよう仕込んでいたのでは？」と疑われる
  - お客様の実カードに合わせることで「デッキは仕込みではない＝本物のマジック」が成立する

### だから音声認識は"トリックの必須条件"ではない

- 音声認識・AIの当ては **④の画面演出（エンタメ／時間つなぎ）専用**
- **AIが外しても物理リビールは崩れない。** むしろ外した方が「デッキは仕込みじゃない」証明になり効果は強まる
- 音声が全滅しても、マジシャンがお客様に直接聞けば物理リビールは成立する（トリックの生命線が技術に依存しない）

### マジシャンへの伝達

- 特別な技術チャネルは不要。マジシャンが戻ったときに **会話で自然にお客様へ確認**すればよい（「で、何のカード選んだの？」）
- AIが正解していれば画面のカードを見て把握してもよい（便宜的手段）

## 3. 演技フロー（マジシャン視点の手順）

1. iPhoneを取り出し「代わりのマジシャンを用意します」→ **スタートボタンを押す**
2. トランプケースに立てかけて離席
3. （以降アプリが自動進行。AIがお客様とやり取りしてカードを"画面で"当てる）
4. 戻ってくる → お客様に本当のカードを確認
5. 「私の手にかかれば余裕」的な締めのあと、**インビジブルデック**で物理ケースからそのカードを出す

## 4. アプリ状態遷移

```
IDLE ──(スタートボタン)──▶ INTRO
INTRO ──(自己紹介台本の再生完了)──▶ LISTEN
LISTEN ──(カード名を認識)──▶ CONFIRM
        └─(認識失敗/タイムアウト)──▶ RETRY ──▶ LISTEN
CONFIRM ──(復唱)──▶ REVEAL
REVEAL ──(派手な演出＋カード面表示)──▶ TAUNT
TAUNT ──(締めの毒舌)──▶ DONE（マジシャン復帰・物理リビール）
DONE ──(リセット操作)──▶ IDLE
```

## 5. 各状態の仕様

### IDLE
- 起動画面。トランプケースの静止画＋「AIマジシャン スリープ中」的な表示
- **普通のスタートボタン**を置く（隠す必要なし）。押すとAIが話し出す

### INTRO（自己紹介）
- 台本を **TTSで再生**
- 例：「初めまして。代理のマジシャンです。担当は人類的な事情でしばらく離席しました。つまらないでしょうが、私が代役を務めます。」

### LISTEN（カード指定）
- 台本：「好きなトランプを1枚、声に出して言ってください。スートと数字。それくらい言えますよね？」
- **音声認識ON**（Web Speech API, `lang=ja-JP`, `interimResults=true`）
- **マイクが動いていることは分かってOK**（隠さない）。「ただのスマホがやっている」感を出す方が、最後の⑥の落差が効く。ただし大げさに強調はしない
- 一定時間（例8秒）無反応 → RETRY

### RETRY（毒舌でリトライ）
- 台本例：「は？聞こえません。もう一度、はっきり。」「滑舌。やり直し。」
- 最大 N 回（例3回）。以降は最後の候補で進める（外れてもトリックは崩れないので気楽でよい）

### CONFIRM（復唱）
- 認識結果を復唱：「ハートの7、ね。はいはい。」
- お客様への確認＋場のテンポづくり（伝達目的ではないので気楽）

### REVEAL（画面内リビール）※できるだけ派手に
- **パチンコの大当たり級の派手さを狙う**（フラッシュ・エフェクト・効果音・振動など）
- トランプケースが弾けて／光ってカードが飛び出す等、テンションが上がる演出
- 出てくるカード面 ＝ 認識・確定したカード
- ここが「スマホすげぇ（＝スマホやから当然）」のピーク。落差づくりの布石

### TAUNT（締め）
- 台本例：「私の手にかかれば、この程度。…担当、まだ戻らないんですか。」
- お客様に「スマホやからできて当然」と思わせる余裕の演出（⑥の落差を作る布石）

### DONE
- マジシャン復帰 → お客様に実カードを確認 → インビジブルデックで物理リビール
- 画面は静かに待機。リセット操作で IDLE に戻す

## 6. 音声認識：カード名パース仕様

認識テキストから **スート**と**数字（ランク）**を抽出する。対象は **52枚（ジョーカーなし）**。

**スートのゆらぎ吸収**
| 正規化 | 認識され得る語 |
|---|---|
| ハート | ハート / はーと |
| スペード | スペード / すぺーど |
| クラブ | クラブ / クローバー / 三つ葉 / みつば |
| ダイヤ | ダイヤ / ダイヤモンド / ダイア |

**ランクのゆらぎ吸収**
| 正規化 | 認識され得る語 |
|---|---|
| A(1) | エース / A / 1 / いち |
| 2〜10 | 数字 / 漢数字 / 読み（しち・なな 等の複数読み対応） |
| J | ジャック / J |
| Q | クイーン / Q |
| K | キング / K |

- 「7」は「しち」「なな」「セブン」等の複数表記を吸収するマッピングを持つ
- スートまたはランクの一方しか取れない場合は RETRY で補完質問（「スートは？」等）
- 認識が外れてもトリックは崩れないため、精度は"エンタメとして許容できる範囲"を目標にする

## 7. TTS（合成音声）仕様

- Web Speech Synthesis API（`SpeechSynthesisUtterance`, `lang=ja-JP`）
- **現状：iOS内蔵音声（Kyoko / Otoya 等）でOK** → オフライン動作
- 機械的な声質はむしろ「AIマジシャン」のキャラとして活かす（`rate` / `pitch` で無機質に寄せる）
- **将来：キャラ（ペルソナ）を差し替え可能に設計する**
  - 例）ツンデレ風女子高生、おじさん風 など
  - 「声（voice/rate/pitch）＋台本セット」を1つのペルソナとしてまとめ、選択できる構造にしておく
- 既知の注意点：
  - iOSは**ユーザー操作起点でないと発話しない** → スタートボタンで解禁
  - `voices` の読み込みは非同期（`voiceschanged` を待つ）
  - iOSで発話が詰まる事象への `resume()` 対策を入れる

## 8. 台本（決め台詞）

> **今後、面白い案を別途ブレストして詰める。** 以下は仮ドラフト。
> ペルソナごとに台本セットを持てる構造にする（v7 参照）。

- INTRO：「初めまして。代理のマジシャンです。担当は人類的な事情で離席中。私が代わりを務めます。」
- LISTEN：「好きなトランプを1枚、声に出して。スートと数字。言えますよね？」
- RETRY：「は？」「聞こえません。もう一度。」「滑舌が課題ですね。やり直し。」
- CONFIRM：「{カード名}、ね。」「はいはい、{カード名}。」
- TAUNT：「私の手にかかればこの程度。」「担当、まだ戻らないんですか。」

## 9. フォールバック / 安全策

- **音声完全不通でも成立**：物理リビールはお客様への口頭確認＋インビジブルデックで完結するため、アプリが失敗してもマジックは崩れない
- **リトライ上限**：毒舌キャラで自然に時間を稼げる（失敗が芸になる）
- アプリはあくまで「つなぎ＆盛り上げ」。トリックの生命線を技術に持たせない

## 10. 難読化方針（CLAUDE.md 準拠）

- 本アプリは「AIマジシャン」自体をお客様に見せる建付けのため、UI・音声認識は隠さない
- 隠すべきは **物理リビールとインビジブルデックの存在**（コード上に手品のネタが出ないようにする）
- localStorage キー／関数名は既存流儀（`_dcc`, `glf()` 等）に合わせ、マジック用語を出さない

## 11. 技術構成

- 純HTML/CSS/JS（フレームワーク不使用・既存アプリと統一）
- Web Speech API（認識）＋ Web Speech Synthesis（合成）
- PWA対応（オフライン動作）
- GitHub Pages ホスティング

## 12. リスク / 要検証

- **騒音下の認識精度**（バーのBGM・ざわつき）→ 実機で要テスト。※外れても致命的でない点は救い
- iOS Safari の音声認識の安定性（`continuous` の途切れ）→ こまめな再起動処理
- TTSの声質・イントネーション（棒読み許容できるか）
- REVEAL演出の派手さ（効果音・バイブ・アニメ）を実機で気持ちよく出せるか

## 13. MVP範囲（フェーズ1）

- スタートボタン → INTRO → LISTEN → CONFIRM → REVEAL → TAUNT の一連自動進行
- カード名パース（52枚）
- TTS再生（iOS内蔵音声）
- REVEAL演出（まずは派手めのエフェクト＋効果音を目標）
- ※物理リビールはマジシャンの実演（インビジブルデック）なのでアプリ実装対象外

フェーズ2以降：ペルソナ差し替え（ツンデレ女子高生・おじさん等）、台本バリエーション、認識精度チューニング、演出の作り込み。
