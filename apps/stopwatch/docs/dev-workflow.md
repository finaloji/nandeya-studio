# 3エージェント開発ハーネス 手順書

ストップウォッチアプリ開発で実践した「Planner → Generator → Evaluator」方式の進め方をまとめたもの。アプリ・ツール開発全般で再利用する。

## 全体の流れ

```
1. 大まかな要望をヒアリング
2. 仕様書を作成（人間と一緒に固める）
3. スプリント単位に分割
4. 各スプリントごとに:
   a. Planner: そのスプリントの詳細仕様を作る
   b. Generator: 仕様に基づいて実装する（worktree隔離）
   c. 人間が確認 → コードレビュー → mainに反映
   d. 実機/ブラウザで動作確認 → フィードバック
5. 全スプリント完了後、UI仕上げ・最終調整
```

## 各エージェントの役割

### 1. ヒアリング（人間 ⇄ Claude本体）

- 最初に1〜4行の要望を聞く
- 曖昧な点を質問で潰す（実装方法ではなく「何を作るか」のレベル）
- 仕様書（docs/xxx-spec.md）としてまとめる
- 仕様が固まるまでコードは書かない

### 2. スプリント分割（Claude本体）

- 仕様書を機能単位の小さなスプリントに分ける
- 1スプリント = 1機能。大きすぎず、検証可能な単位にする
- 表形式で進捗を管理（完了/作業中/未着手）

### 3. Planner（Agent: subagent_type="Plan"）

- 役割: スプリントの詳細仕様を「何が起きるか」レベルで展開する
- **技術的な実装詳細（DB設計、関数名など）には踏み込ませない**
  - 理由: Plannerが実装方法を間違えると下流（Generator）に伝播するため
- 入力: スプリントの概要、現在のコードの状態
- 出力: 詳細な製品仕様書（状態遷移、UI、データの扱いなど）

### 4. Generator（Agent: isolation="worktree"）

- 役割: Plannerの仕様を実際のコードに実装する
- 必ず `isolation: "worktree"` で隔離実行する（本体に影響を与えない）
- プロンプトには既存コードの構造（変数名・関数名）を明記し、一貫性を保たせる
- 完了後、Claude本体がworktreeの差分を読んでレビューする

### 5. 人間によるレビュー・動作確認

- Claude本体がworktreeのコードをReadで確認
- 問題なければmainブランチにコピー → コミット → push
- 実機（iPhone等）やブラウザで動作確認してもらう
- フィードバックがあれば、その場で直接修正 or 次のサイクルへ

## 実践のコツ

- **Plannerに実装詳細を書かせない**: 「SQLiteのテーブル構成」のような指定は禁止。仕様が技術選定を縛ると、Generatorの自由度が下がり、下流で帳尻合わせが必要になる
- **Generatorへの指示は具体的に**: 既存の変数名・関数名・コード構造を明記する。そうしないとGeneratorが命名規則を変えてしまい、スプリント間で一貫性が崩れる
- **小さいスプリントに分割する**: 1機能ずつ検証できるサイズにすることで、問題があってもどこが原因か特定しやすい
- **worktree隔離は必須**: Generatorの実装ミスが本体のコードに直接影響しないようにする
- **実機確認は早めに**: ブラウザのプレビューツールが使えない環境（サーバー起動不可など）もあるため、GitHub Pages等で実機から直接確認できる経路を早めに作る

## このプロジェクトでの実例

ストップウォッチアプリ（`stopwatch.html`）を以下のスプリントで開発した。

| Sprint | 内容 |
|--------|------|
| 1 | 基本UI（タブバー＋画面の見た目） |
| 2 | 開始・停止・リセットの動作 |
| 3 | ラップ機能 |
| 4 | コア機能（コア計算ロジック） |
| 5 | 隠し設定画面 |
| 6 | PWA対応（アイコン・manifest） |
| 7 | UI仕上げ |
| 追加 | 開始トリガー設定の追加機能 |

各スプリントで Planner → Generator → 人間レビュー → 実機確認 のサイクルを回した。

## コマンド化

`/sprint` コマンドでこのフローを呼び出せる（`~/.claude/commands/sprint.md` に定義）。
