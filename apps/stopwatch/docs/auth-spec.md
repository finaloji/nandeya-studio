# ID・パスコード認証＋端末ロック 仕様書

## 概要

購入者（マジシャン）のみがストップウォッチアプリを使えるようにする。ID・パスコードでログインし、1つのIDにつき1台の端末でしか使用できないようにする。Firebase（Firestore）を使ってサーバーレスで実現する。

## Firestoreのデータ構造

### コレクション: `licenses`

ドキュメントID = 発行したID（例: `customer001`）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| passcode | string | パスコード |
| deviceId | string \| null | 登録済み端末の識別子。未登録ならnull |
| createdAt | timestamp | 発行日時 |
| note | string | お客様名などのメモ（任意） |

### コレクション: `deviceChangeRequests`

ドキュメントID = 自動採番

| フィールド | 型 | 説明 |
|-----------|-----|------|
| licenseId | string | 対象のID |
| requestedAt | timestamp | 申請日時 |
| status | string | `pending` / `approved` |

## 機能1: stopwatch.html のログインゲート

### 起動時の流れ

1. アプリ起動時、まずローカル（localStorage）に「ログイン済み＋端末ID」の記録があるか確認する
2. 記録があれば、Firestoreにその端末IDが今も有効か確認してから通常画面を表示する
3. 記録がなければ、ログイン画面を表示する

### ログイン画面

- ID入力欄
- パスコード入力欄
- 「ログイン」ボタン
- デザインは黒背景でアプリの雰囲気に合わせる

### ログイン処理

1. 入力されたID・パスコードでFirestoreの`licenses`を検索
2. 該当IDが存在しない、またはパスコードが一致しない → エラー表示
3. 一致した場合:
   - `deviceId`が`null`（未登録）の場合 → この端末の識別子を生成してFirestoreに登録 → ログイン成功
   - `deviceId`が既に登録されている場合:
     - 今回の端末の識別子と一致する → ログイン成功（同じ端末からの再ログイン）
     - 一致しない → 「他の端末で使用中です」エラーを表示し、ログイン拒否

### 端末識別子の生成方法

- ブラウザ生成のランダムなUUIDを作り、localStorageに保存する（初回アクセス時に1度だけ生成）
- この値を「端末ID」として扱う

### 端末変更リクエスト機能

ログイン画面に「この端末を新しく登録したい場合」という案内とボタンを設置する。

1. お客様がID・パスコードを入力した状態で「端末変更をリクエスト」ボタンを押す
2. ID・パスコードが正しければ、Firestoreの`deviceChangeRequests`に申請を追加（status: pending）
3. 「申請を送信しました。承認をお待ちください」という案内を表示
4. 自分がadmin.htmlで承認すると、該当licenseの`deviceId`が`null`にリセットされる
5. お客様は再度ログイン画面からログインし直すと、新しい端末が登録される

## 機能2: admin.html（管理画面）

### アクセス制限

- admin.html自体に簡易パスワードを設定する（自分専用なので固定パスワード1つでよい）
- パスワードはコード内にハッシュ化して埋め込む

### 画面構成

1. **ID発行フォーム**
   - ID（任意の文字列、重複チェックあり）
   - パスコード（自動生成 or 手入力）
   - メモ（お客様名など、任意）
   - 「発行」ボタン

2. **ID一覧テーブル**
   - ID / パスコード / 登録端末の有無 / メモ / 発行日
   - 各行に「削除」ボタン（ライセンス無効化）
   - 各行に「端末登録をリセット」ボタン（手動でdeviceIdをnullに戻す）

3. **端末変更リクエスト一覧**
   - pending状態の申請を一覧表示
   - 「承認」ボタンを押すと該当ライセンスのdeviceIdをnullにし、status を approved に更新

## Firestoreセキュリティルール

クライアントから直接Firestoreにアクセスするため、ルールでの保護が重要。

- `licenses`コレクション: パスコード照合のための読み取りのみ許可（書き込みは制限）。実際の認証ロジック上、書き込みもクライアントから必要になる箇所があるため、フィールド単位での制御を検討する
- 基本方針: 読み取りは許可、`deviceId`の書き込みは「現在nullの場合のみ」「または値が一致する場合のみ」許可するルールにする
- admin.html側の操作（発行・削除・リセット）は別途、管理パスワード確認後のみ実行可能にする（Firestoreルール上は区別できないため、UI側の制御＋ルールでの最低限の制約を組み合わせる）

## 注意事項

- お客様用のID・パスコードは平文でFirestoreに保存される設計（パスコード自体がそこまで機密性の高い情報ではなく、小規模・信頼ベース運用のため簡素化している）
- 完全な不正利用防止ではなく、「気軽なコピー・転用を防ぐ」レベルの抑止力として設計している
