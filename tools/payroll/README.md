# 勤怠・給与計算補助システム（payroll）

> 作成日: 2026-07-22
> ステータス: 設計ドラフト（レビュー中・未実装）

## 1. 目的

株式会社なんで屋の給与計算（毎月約2時間）を効率化する。

**このシステムがやること**

- スタッフの打刻を集める
- 勤務時間を自動計算する
- 時給・交通費・集客バック・コミッションを合算して支給額を出す
- 給与ソフトに渡すCSVと、本人向けの明細PDFを出す

**このシステムがやらないこと**

- 所得税・社会保険料・住民税の計算（給与ソフトの担当）
- 年末調整、法定調書
- 振込処理

税金まわりは一切触らない。**給与ソフトの手前までを担う道具** と位置づける。

## 2. 対象と規模

| 項目 | 内容 |
|------|------|
| 対象人数 | 6〜20人 |
| 雇用形態 | 時給制のアルバイト、日給制のマジシャン（イリュージョンランド）、業務委託のマジシャン（混在） |
| 店舗 | 3店舗（マジックバーなんで屋！？本店 / マジックバーなんで屋！？東門街店 / イリュージョンランドなんで屋！？） |
| 掛け持ち | あり。**時給は人に紐づき、店舗では変わらない** |
| メイン端末 | スマホ（PCも利用可のレスポンシブ） |

## 3. 技術構成

素のHTML / CSS / JavaScript ＋ Firebase Firestore ＋ GitHub Pages。
リポジトリ内の他ツールと同じ構成に揃える。

### なぜ Next.js / Supabase / Vercel を採らなかったか

当初 Next.js + TypeScript + Supabase + Vercel を検討したが、以下の理由で見送った。

- **権限とロックはFirestoreで足りる** — 「権限分離・確定後ロック・修正履歴」はFirestoreセキュリティルールでサーバー側に強制できる。サーバーコードを持つ必要がない
- **規模が小さい** — 6〜20人・月1回。Postgresの集計・トランザクションの優位性は、この規模ではコードで吸収できる。一方ビルド・デプロイ・マイグレーションの運用コストは毎回かかる
- **リポジトリ内の他アプリと構成が揃う** — ビルド工程を持たない静的HTMLで統一されており、1つだけ別構成にすると保守の対象が二重になる

> **補足（2026-07-22 追記）:** 当初はこれに加えて「請求ツールのFirestoreに案件データがあるので同じDBに置きたい」という理由を挙げていた。
> 請求ツールが未稼働のため連携をスコープ外にした結果、**この理由は失効している。**
> 残る3点でもFirestoreを推す判断は変えていないが、当初より根拠が弱まっていることは記録しておく。
> 将来サブスクプラットフォーム（`docs/`）でNext.jsに寄せる判断をした場合は、その時点で移行を検討する。

### Firebaseプロジェクト

**payroll 専用のFirebaseプロジェクトを新規作成する。**

給与データは機密性が高く、他ツールとセキュリティルールを同居させたくない。
請求ツールとの連携をスコープ外にしたため、プロジェクトを分けるデメリットもない。

### 認証

- 管理者・社員: 共通パスコード ＋ 個人選択
- スタッフ: 個人ごとの打刻用URL（トークン付き）。パスコード入力なしで打刻画面に入れる

> スタッフに毎回パスコードを打たせると打刻されなくなる。打刻のハードルは限界まで下げる。

## 4. 権限設計

| ロール | 打刻 | 勤怠の閲覧 | 打刻の修正・承認 | 給与一覧・CSV | 明細PDF | 月次確定・ロック解除 | マスタ編集 |
|---|---|---|---|---|---|---|---|
| **管理者**（橋本） | ○ | 全店舗 | ○ | ○ | 全員分 | ○ | ○ |
| **社員** | ○ | 全店舗 | ○ | ○ | 全員分 | ✕ | ✕ |
| **スタッフ** | ○ | 自分のみ | ✕（申請のみ） | ✕ | 自分のみ | ✕ | ✕ |

- 社員は全店舗を修正・承認できる。店舗による閲覧制限は設けない
- **月次確定とロック解除は管理者のみ**。ここが金額の最終責任点
- スタッフは自分の打刻について「修正申請」を出せる（直接編集はできない）

Firestoreセキュリティルールでサーバー側に強制する。クライアント側のUI制御だけに頼らない。

## 7. 給与計算ロジック

### 7-1. 1日の労働時間

```
休憩合計 = Σ(休憩終了 - 休憩開始)
実労働時間 = (退勤 - 出勤) - 休憩合計
```

- 未退勤の日は計算対象外とし、勤怠一覧で警告表示する
- 休憩の開始のみで終了がない場合も同様に警告
- **深夜割増・残業割増は計算しない**（現状発生していないため。必要になった時点で追加する）

### 7-2. 丸め

**30分単位・月合計で丸める。**

日ごとの労働時間は1分単位のまま保持し、月の合計を出した最後に一度だけ30分単位へ丸める。
日ごとに丸めてから足すと、丸め誤差が出勤日数分だけ積み上がるため採らない。

```
月間労働時間（生） = Σ 日ごとの実労働時間（1分単位）
月間労働時間（丸め後） = 30分単位に丸めた値
```

丸めの方向（切り捨て／切り上げ／30分未満切り捨て・30分以上切り上げ）は `payroll_settings.roundingMode` で設定する。
**初期値は「30分未満切り捨て・30分以上切り上げ」**（＝30分単位の四捨五入）とする。

> 労働時間の端数処理は、常に切り捨てるとスタッフに一方的に不利になる。
> 一律切り捨てで運用している場合は、社労士に確認することをおすすめする（私は労務の専門家ではないため断定はしない）。
> 現在どの方向で処理しているかは要確認（下記「未確定事項」参照）。

### 7-3. 月次の支給額

```
基本給（時給制） = 時給 × 月間労働時間（丸め後）
基本給（日給制） = 日給 × 出勤日数
基本給（業務委託）= 0

集客バック   = 月間集客人数 × 400円
交通費       = Σ 日ごとの交通費（下記）
コミッション = Σ payroll_commissions.amount
──────────────────────────────
総支給額     = 基本給 + 集客バック + 交通費 + コミッション + 手動調整
```

- **店舗内でのマジック出演は時給・日給に含まれる**ため、別途の出演料は計上しない
- 掛け持ちしても時給は同じなので、店舗ごとに分けず月で合算する（店舗別の集計は表示のみ）
- 業務委託のマジシャンは実質コミッションのみで支給額が決まる

#### 交通費の計算

日ごとに `transportType` を見て支給額を決める。

| `transportType` | 支給額 | 使う場面 |
|---|---|---|
| `round`（既定） | 片道額 × 2 | 通常の出勤 |
| `oneWay` | 片道額 × 1 | **帰りを車で送った日**（行きの分だけ支給） |
| `none` | 0 | 往復とも送迎した日 |

- 打刻時は常に `round` で記録される。スタッフには選ばせない
- **送った日は、勤怠一覧で社員が `oneWay` に切り替える。** 承認作業のついでに直せる位置に置く
- 片道額をマスタに持つのは、`round` を2倍で導ける一方、往復額からの2分割は端数が出るため

#### 日給制の扱い（イリュージョンランド）

- 出勤日数のカウントは打刻から取る（1日1件の `payroll_attendances` ＝ 1日）
- **労働時間は記録するが、支給額には使わない**。勤怠一覧では時間も表示する（実態把握のため）
- 丸めは日給制には適用しない
- 時給制と日給制の切り替えは `payroll_staff.employmentType` の1箇所で決まる

### 7-4. コミッションの計算方式

| 方式 | 計算 | 用途の例 |
|---|---|---|
| 固定額 `fixed` | `fixedAmount` | 1現場いくらの出演料 |
| 売上割合 `rate` | `baseAmount × rate` | 外部案件の出演料歩合、営業の成約歩合 |
| 数量単価 `unit` | `quantity × unitPrice` | 物販の販売数 × 単価 |

いずれも計算結果は `amount` の**初期値**であり、画面上で上書きできる。上書きした場合は操作履歴に残す。

### 7-5. 外部案件との連携（スコープ外）

請求ツール（`tools/invoice/`）は現時点で未稼働のため、**案件データからのコミッション自動生成は行わない。**
コミッションはすべて手入力とする。

請求ツールの運用が始まった段階で、案件（請求済・入金済み）から担当マジシャン別の売上を引いて
コミッション候補を自動生成する機能を検討する。その際も自動確定はせず、候補提示に留める。

### 7-6. 確定とロック

- 月次確定すると `payroll_monthly.status = confirmed` になる
- 確定後は、その月の打刻・コミッション・月次データが**すべて編集不可**になる（Firestoreルールで強制）
- ロック解除は管理者のみ。解除は操作履歴に必ず残る

## 8. 出力

### 8-1. スプレッドシート向け出力

現在の給与計算は**スプレッドシートへの手入力**で行っている。市販の給与ソフトの取り込み仕様に
縛られないため、出力形式は**今のシートに合わせて自由に決められる**。

3つの受け渡し方法を用意する（実装は上から順に）。

| 方法 | 内容 | 実装コスト |
|---|---|---|
| **① コピー用テキスト** | 画面のボタンでTSVをクリップボードにコピー → シートに貼るだけ | 小 |
| **② CSVダウンロード** | ファイルとして落として取り込む | 小 |
| **③ シートへ直接書き込み** | Google Sheets API で該当月のシートを自動更新 | 中 |

**①を最優先で作る。** ファイル経由より速く、スマホからでも使え、追加の認証設定も要らない。
③は毎月の手入力を完全に消せるが、Google認証（OAuth／サービスアカウント）が必要になり、
静的ホスティングのままでは実現しにくい。第2弾を運用してから、必要性を見て判断する。

出力する列（1行1スタッフ）:

```
スタッフ名, 労働時間, 出勤日数, 基本給, 集客人数, 集客バック, 交通費, コミッション, 調整額, 総支給額
```

#### 現在のシートとの関係

現在のシートは **日付が行・人が列のマトリクス**（勤怠表の形）になっている。
この形は「勤怠一覧」画面がそのまま置き換えるため、**シート側に勤怠を持つ必要がなくなる。**

シートを作り直すなら、役割を控除計算だけに絞り、**1行1スタッフの縦持ち**にするのが素直。

```
スタッフ名 | 総支給額 | 源泉徴収 | 社会保険料 | 差引支給額
           ↑ここまでシステムが出力    ↑ここから先はシート側の計算
```

マトリクスのままだと、システムの出力（縦持ち）を貼るのに毎回転置が要る。
シートの作り直しは本システムの実装対象外だが、**第2弾の出力形式を決める時点で並行して進めるのがよい。**

> **控除・手取りの計算はこのシステムでは行わない。** スプレッドシート側に残っている
> 源泉徴収・社会保険料の計算は、そのまま手元に置く。このシステムは
> 「支給額の列までを自動で埋める」ところまでを担当する。

### 8-2. 明細PDF

- 人別・月次。プレビュー表示 → ブラウザの印刷機能でPDF保存（A4縦）
- 請求ツールと同じ方式（外部ライブラリ不使用）
- 記載内容:
  - 支給年月、氏名、会社名
  - 勤怠（労働時間・出勤日数・店舗別内訳）
  - 支給（基本給・集客バック・交通費・コミッション・調整額・総支給額）
  - コミッション明細（種別・内容・金額の行）
- **控除・手取りは記載しない**（給与ソフト側の担当のため）

## 9. 開発手順

一度に全部作らず、4段階に分ける。各段階を実際に1ヶ月運用してから次へ進む。

### 第1弾 — 打刻と勤怠（効果が最も大きい）

- スタッフ／店舗マスタ、打刻用URL発行

## 10. 未確定事項

実装前に決める必要があるもの。

第1弾（打刻・勤怠）の実装には影響しない。第2弾までに決める。

1. **丸めの方向** — 30分単位・月合計は確定。切り捨てか、30分未満切り捨て・以上切り上げか
2. **日給制マジシャンの範囲** — イリュージョンランド以外にも日給の人がいるか。日給の人が他店に応援に行った場合はどう払うか
3. **業務委託マジシャンの扱い** — 店舗シフトに入ることがあるか。あるなら時給が発生するのか、出演のみか
4. **集客人数の申告の裏取り** — 自己申告のみで運用できるか、社員が事後に確認する運用を入れるか
5. **締め日** — 月末締めか、20日締めなどか
6. **作り直し後のスプレッドシートの形** — 8-1の縦持ち案でよいか
