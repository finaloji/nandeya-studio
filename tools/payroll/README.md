# 勤怠・給与計算補助システム（payroll）

> 作成日: 2026-07-22
> ステータス: 設計ドラフト（レビュー中・未実装）

## 1. 目的

株式会社なんで屋の給与計算（毎月約2時間）を効率化する。

**このシステムがやること**

- スタッフの打刻を集める
- 勤務時間を自動計算する
- 時給・交通費・集客バック・コミッションを合算して支給額を出す
- 給与ソフトに渡すCSVと、本人向けの明細PDFを出す

**このシステムがやらないこと**

- 所得税・社会保険料・住民税の計算（給与ソフトの担当）
- 年末調整、法定調書
- 振込処理

税金まわりは一切触らない。**給与ソフトの手前までを担う道具** と位置づける。

## 2. 対象と規模

| 項目 | 内容 |
|------|------|
| 対象人数 | 6〜20人 |
| 雇用形態 | 時給制のアルバイト、業務委託のマジシャン（混在） |
| 店舗 | 3店舗（マジックバーなんで屋！？本店 / マジックバーなんで屋！？東門街店 / イリュージョンランドなんで屋！？） |
| 掛け持ち | あり。**時給は人に紐づき、店舗では変わらない** |
| メイン端末 | スマホ（PCも利用可のレスポンシブ） |

## 3. 技術構成

素のHTML / CSS / JavaScript ＋ Firebase Firestore ＋ GitHub Pages。
リポジトリ内の他ツールと同じ構成に揃える。

### なぜ Next.js / Supabase / Vercel を採らなかったか

当初 Next.js + TypeScript + Supabase + Vercel を検討したが、以下の理由で見送った。

- **データが分断する** — 請求ツール（`tools/invoice/`）のFirestoreに「マジシャン一覧・料金マスタ・案件（担当マジシャン・売上）」が既にある。コミッションの元データはまさにそれ。別DBに置くと案件の二重入力か、2DB同期が発生する。効率化が目的なのに手作業が増える
- **権限とロックはFirestoreで足りる** — 「権限分離・確定後ロック・修正履歴」はFirestoreセキュリティルールでサーバー側に強制できる。サーバーコードを持つ必要がない
- **規模が小さい** — 6〜20人・月1回。Postgresの集計・トランザクションの優位性は、この規模ではコードで吸収できる。一方ビルド・デプロイ・マイグレーションの運用コストは毎回かかる

将来サブスクプラットフォーム（`docs/`）でNext.jsに寄せる判断をした場合は、その時点で移行を検討する。Firestoreのデータ構造は下記のとおり正規化寄りにしてあるので、移行は不可能ではない。

### Firebaseプロジェクト

**請求ツールと同じプロジェクトを共用する。** マジシャン一覧・案件データを直接読むため。
（ストップウォッチ等のマジックアプリ用プロジェクトとは分離したまま）

### 認証

- 管理者・社員: 共通パスコード ＋ 個人選択
- スタッフ: 個人ごとの打刻用URL（トークン付き）。パスコード入力なしで打刻画面に入れる

> スタッフに毎回パスコードを打たせると打刻されなくなる。打刻のハードルは限界まで下げる。

## 4. 権限設計

| ロール | 打刻 | 勤怠の閲覧 | 打刻の修正・承認 | 給与一覧・CSV | 明細PDF | 月次確定・ロック解除 | マスタ編集 |
|---|---|---|---|---|---|---|---|
| **管理者**（橋本） | ○ | 全店舗 | ○ | ○ | 全員分 | ○ | ○ |
| **社員** | ○ | 全店舗 | ○ | ○ | 全員分 | ✕ | ✕ |
| **スタッフ** | ○ | 自分のみ | ✕（申請のみ） | ✕ | 自分のみ | ✕ | ✕ |

- 社員は全店舗を修正・承認できる。店舗による閲覧制限は設けない
- **月次確定とロック解除は管理者のみ**。ここが金額の最終責任点
- スタッフは自分の打刻について「修正申請」を出せる（直接編集はできない）

Firestoreセキュリティルールでサーバー側に強制する。クライアント側のUI制御だけに頼らない。

## 5. 画面一覧

### スタッフ向け

| 画面 | 内容 |
|---|---|
| **打刻** | 大きなボタン1つ（状態により 出勤／休憩開始／休憩終了／退勤 が切り替わる）。出勤時に店舗を選択。**退勤時に「今日の集客人数」を入力** |
| **自分の勤怠** | 当月の打刻一覧。誤りがあれば修正申請を出す |
| **自分の明細** | 確定後の給与・コミッション明細を閲覧・PDF保存 |

### 管理者・社員向け

| 画面 | 内容 |
|---|---|
| **勤怠一覧** | 月×店舗で打刻を一覧。未退勤・異常値を強調。セルを直接修正、まとめて承認 |
| **修正申請** | スタッフからの申請を承認／却下 |
| **コミッション登録** | 出演料・営業・物販・アシスタント料を登録。固定額／売上割合／数量単価を選択 |
| **給与一覧** | 全員の当月支給額を一覧。項目ごとの内訳を確認、手動調整、CSV出力、月次確定 |
| **明細出力** | 人別の明細PDF（プレビュー→ブラウザ印刷でPDF保存。請求ツールと同方式） |
| **マスタ管理** | スタッフ（時給・交通費・ロール）、店舗、コミッション種別の編集。**管理者のみ** |
| **操作履歴** | 修正履歴の閲覧 |

## 6. データベース設計（Firestore）

### `payroll_staff` — スタッフ

| フィールド | 型 | 内容 |
|---|---|---|
| `name` | string | 氏名 |
| `role` | string | `admin` / `employee` / `staff` |
| `employmentType` | string | `hourly`（時給）/ `contract`（業務委託） |
| `hourlyWage` | number | 時給（円）。業務委託は0 |
| `transportAllowance` | number | 交通費（1出勤あたりの定額・円） |
| `magicianId` | string? | 請求ツールのマジシャンID。外部案件の歩合を紐づけるため |
| `punchToken` | string | 打刻用URLのトークン |
| `active` | boolean | 在籍中か |

### `payroll_stores` — 店舗

| フィールド | 型 | 内容 |
|---|---|---|
| `name` | string | 店舗名 |
| `order` | number | 表示順 |
| `active` | boolean | 営業中か |

初期データ: 本店 / 東門街店 / イリュージョンランド

### `payroll_attendances` — 打刻（1日1スタッフ1件）

| フィールド | 型 | 内容 |
|---|---|---|
| `staffId` | string | スタッフID |
| `storeId` | string | 勤務店舗 |
| `workDate` | string | 勤務日 `YYYY-MM-DD`（日跨ぎは出勤日に属させる） |
| `clockIn` | timestamp | 出勤 |
| `clockOut` | timestamp? | 退勤 |
| `breaks` | array | `[{ start, end }]` 休憩は複数回を許す |
| `guestCount` | number | 集客人数（退勤時の自己申告） |
| `workMinutes` | number | 実労働時間（分）。承認時に確定 |
| `breakMinutes` | number | 休憩合計（分） |
| `status` | string | `open`（勤務中）/ `submitted`（退勤済・未承認）/ `approved`（承認済） |
| `note` | string | 備考 |

> 日跨ぎ勤務（深夜営業）を扱うため、`workDate` は打刻時刻から自動導出せず **出勤ボタンを押した営業日** を持たせる。

### `payroll_correction_requests` — 修正申請

| フィールド | 型 | 内容 |
|---|---|---|
| `attendanceId` | string | 対象の打刻 |
| `staffId` | string | 申請者 |
| `requestedChanges` | map | 希望する値 |
| `reason` | string | 理由 |
| `status` | string | `pending` / `approved` / `rejected` |

### `payroll_commissions` — コミッション

| フィールド | 型 | 内容 |
|---|---|---|
| `staffId` | string | 対象スタッフ |
| `yearMonth` | string | `YYYY-MM` |
| `category` | string | `performance`（出演料）/ `sales`（営業）/ `merch`（物販）/ `assistant`（アシスタント料） |
| `calcType` | string | `fixed`（固定額）/ `rate`（売上割合）/ `unit`（数量単価） |
| `baseAmount` | number? | `rate` の場合の売上額 |
| `rate` | number? | `rate` の場合の割合（例 0.3） |
| `quantity` | number? | `unit` の場合の数量 |
| `unitPrice` | number? | `unit` の場合の単価 |
| `fixedAmount` | number? | `fixed` の場合の金額 |
| `amount` | number | **確定金額**。自動計算した値を入れるが、手で上書きできる |
| `invoiceCaseId` | string? | 請求ツールの案件ID（外部案件から引いた場合） |
| `note` | string | 備考 |

> 「大体は決まっているがこちらで調整できるようにしたい」という要望に対応するため、
> 計算パラメータと確定金額 `amount` を**別々に持つ**。自動計算は初期値であり、最終値ではない。

### `payroll_monthly` — 月次給与（1人1ヶ月1件）

| フィールド | 型 | 内容 |
|---|---|---|
| `staffId` | string | 対象スタッフ |
| `yearMonth` | string | `YYYY-MM` |
| `workMinutes` | number | 月間労働時間（分） |
| `workDays` | number | 出勤日数 |
| `basePay` | number | 基本給 |
| `guestCountTotal` | number | 月間集客人数 |
| `guestBonus` | number | 集客バック |
| `transportTotal` | number | 交通費合計 |
| `commissionTotal` | number | コミッション合計 |
| `adjustment` | number | 手動調整額（±） |
| `adjustmentNote` | string | 調整理由 |
| `grossPay` | number | **総支給額** |
| `status` | string | `draft` / `confirmed`（確定・ロック済） |
| `confirmedAt` / `confirmedBy` | timestamp / string | 確定した日時と人 |

### `payroll_audit_logs` — 操作履歴

| フィールド | 型 | 内容 |
|---|---|---|
| `at` | timestamp | 日時 |
| `actorId` | string | 操作した人 |
| `action` | string | `punch` / `edit_attendance` / `approve` / `edit_commission` / `confirm_month` / `unlock_month` |
| `targetType` / `targetId` | string | 対象 |
| `before` / `after` | map | 変更前後の値 |

打刻の修正、コミッションの変更、月次確定・ロック解除を記録する。**削除も追記で残す**（物理削除しない）。

### `payroll_settings` — 設定

| フィールド | 型 | 内容 |
|---|---|---|
| `guestBonusUnitPrice` | number | 集客バック単価（初期値 400） |
| `roundingUnit` | number | 労働時間の丸め単位（分） |
| `roundingTiming` | string | `daily`（日ごと）/ `monthly`（月合計） |

## 7. 給与計算ロジック

### 7-1. 1日の労働時間

```
休憩合計 = Σ(休憩終了 - 休憩開始)
実労働時間 = (退勤 - 出勤) - 休憩合計
```

- 未退勤の日は計算対象外とし、勤怠一覧で警告表示する
- 休憩の開始のみで終了がない場合も同様に警告
- **深夜割増・残業割増は計算しない**（現状発生していないため。必要になった時点で追加する）

### 7-2. 丸め

労働時間の丸め単位・タイミングは `payroll_settings` で設定する。
初期値は **1分単位・丸めなし** とし、実運用を見て調整する。

> 丸めは日ごとに行うか月合計で行うかで金額が変わる。日ごと切り捨てはスタッフに不利になりやすい。
> 現在の運用がどちらか確認した上で確定させる（下記「未確定事項」参照）。

### 7-3. 月次の支給額

```
基本給       = 時給 × 月間労働時間
集客バック   = 月間集客人数 × 400円
交通費       = 出勤日数 × 交通費（1出勤あたり定額）
コミッション = Σ payroll_commissions.amount
──────────────────────────────
総支給額     = 基本給 + 集客バック + 交通費 + コミッション + 手動調整
```

- **店舗内でのマジック出演は時給に含まれる**ため、別途の出演料は計上しない
- 掛け持ちしても時給は同じなので、店舗ごとに分けず月で合算する（店舗別の集計は表示のみ）
- 業務委託のマジシャンは時給0のため、実質コミッションのみで支給額が決まる

### 7-4. コミッションの計算方式

| 方式 | 計算 | 用途の例 |
|---|---|---|
| 固定額 `fixed` | `fixedAmount` | 1現場いくらの出演料 |
| 売上割合 `rate` | `baseAmount × rate` | 外部案件の出演料歩合、営業の成約歩合 |
| 数量単価 `unit` | `quantity × unitPrice` | 物販の販売数 × 単価 |

いずれも計算結果は `amount` の**初期値**であり、画面上で上書きできる。上書きした場合は操作履歴に残す。

### 7-5. 外部案件との連携

請求ツールの案件データ（ステータスが「請求済」または「入金済み」）から、
担当マジシャン別の売上を引いてコミッション候補を自動生成する。

```
案件（担当マジシャン=まさゆき, 公演費 50,000, 請求済）
   ↓
コミッション候補（staffId=まさゆき, category=performance, calcType=rate, baseAmount=50,000）
   ↓ 画面で率を確認・調整して確定
payroll_commissions に登録
```

**自動で確定はしない。** 候補として提示し、人が確認して登録する。

### 7-6. 確定とロック

- 月次確定すると `payroll_monthly.status = confirmed` になる
- 確定後は、その月の打刻・コミッション・月次データが**すべて編集不可**になる（Firestoreルールで強制）
- ロック解除は管理者のみ。解除は操作履歴に必ず残る

## 8. 出力

### 8-1. 給与計算用CSV

1行1スタッフ。給与ソフトへの取り込み用。

```
スタッフ名, 労働時間, 出勤日数, 基本給, 集客人数, 集客バック, 交通費, コミッション, 調整額, 総支給額
```

> 列の並びと項目名は、実際に使っている給与ソフトの取り込み仕様に合わせる必要がある。
> ソフト名が未確認のため、暫定でこの並びにしている（下記「未確定事項」参照）。

### 8-2. 明細PDF

- 人別・月次。プレビュー表示 → ブラウザの印刷機能でPDF保存（A4縦）
- 請求ツールと同じ方式（外部ライブラリ不使用）
- 記載内容:
  - 支給年月、氏名、会社名
  - 勤怠（労働時間・出勤日数・店舗別内訳）
  - 支給（基本給・集客バック・交通費・コミッション・調整額・総支給額）
  - コミッション明細（種別・内容・金額の行）
- **控除・手取りは記載しない**（給与ソフト側の担当のため）

## 9. 開発手順

一度に全部作らず、4段階に分ける。各段階を実際に1ヶ月運用してから次へ進む。

### 第1弾 — 打刻と勤怠（効果が最も大きい）

- スタッフ／店舗マスタ、打刻用URL発行
- 打刻画面（出勤・休憩開始・休憩終了・退勤・集客人数）
- 勤怠一覧、打刻修正、承認
- 修正申請
- 操作履歴

> ここまでで集計作業がまるごと消える。2時間のうち最も大きい塊。

### 第2弾 — 基本給と支給一覧

- 時給・交通費のマスタ
- 月次の自動計算（基本給・集客バック・交通費）
- 給与一覧、手動調整
- CSV出力

### 第3弾 — コミッション

- コミッション登録画面（3方式）
- 請求ツールの案件からの候補自動生成
- 月次への合算

### 第4弾 — 明細と確定

- 明細PDF
- 月次確定・ロック
- スタッフ向けの明細閲覧画面

実装は `/sprint`（Planner → Generator → 人間レビュー）で進める。
進捗は `docs/plan.md` で管理する（本READMEには進捗を書かない）。

## 10. 未確定事項

実装前に決める必要があるもの。

1. **給与ソフトの名称と取り込みCSVの仕様** — 8-1の列定義がこれで決まる
2. **労働時間の丸め** — 現在の運用は1分単位か15分単位か。日ごとに丸めるか月合計か
3. **交通費** — 1出勤あたりの定額で問題ないか（実費精算や日によって変わるケースはないか）
4. **業務委託マジシャンの扱い** — 店舗シフトに入ることがあるか。あるなら時給が発生するのか、出演のみか
5. **集客人数の申告の裏取り** — 自己申告のみで運用できるか、店長が事後に確認する運用を入れるか
6. **締め日** — 月末締めか、20日締めなどか
